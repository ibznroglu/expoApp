# Leaderboard V1 — Implementation Plan

**Date:** 2026-06-23
**Status:** READY
**Scope source:** `thoughts/shared/backlog.md` (section "V1 leaderboard decisions"), supersedes the deferred V2 in `thoughts/shared/plans/leaderboard-spec.md`.

---

## Goal

Data model + service + score submission + profile chips ONLY. The leaderboard screen UI is a separate follow-up task.

- A single Appwrite collection `userStats` holding per-user aggregate stats.
- Score persistence on quick-game completion for **verified** users only (guests are ephemeral — never persisted, never ranked).
- Query-time ranking by `totalScore DESC` (no Appwrite Function).
- Real data wired into the profile "Skor" and "Doğruluk" chips.
- Per-document ownership from day one: each `userStats` doc is world-readable but only its owner may update/delete it.

### Resolved decisions

1. Empty-state / missing-value text: `—` (em-dash). Not "Yakında".
2. Accuracy format: `%85` — Turkish convention, percent sign **before** the number, whole number, no decimals.
3. Permissions: **per-doc ownership from the start**. Collection has Document Security ON; collection-level create permission `users/verified`; each doc gets document-level `[read(any), update(user(userId)), delete(user(userId))]` set at CREATE time. `updateDocument` passes no permissions (they persist from create).
4. `getLeaderboard` default limit: `50` with `offset` param — built into scoreService for the future screen; no screen built in this plan.
5. Score = 0 games: not submitted (guarded in quick-game). Verified flag is unreliable on this project (per backlog); the real security gate is the server-side `users/verified` collection permission — the client `isGuest` check is a UX optimization only.

## Data Model — Appwrite collection `userStats`

One document per user. Document `$id` generated by Appwrite; `userId` is the queryable identity key.

| Attribute | Type | Notes |
|-----------|------|-------|
| `userId` | string (size 64), required | Appwrite account `$id`. |
| `userName` | string (size 128), required | Snapshot of display name, refreshed on every write. |
| `totalScore` | integer, required, default 0 | Running sum of per-game scores (0–100 per game, multiples of 10). |
| `gamesPlayed` | integer, required, default 0 | Count of persisted games. |
| `bestScore` | integer, required, default 0 | Max single-game score. |
| `totalCorrect` | integer, required, default 0 | Sum of correct answers across games. |
| `totalQuestions` | integer, required, default 0 | Sum of questions answered across games (10 per game). |
| `lastPlayedAt` | datetime, required | ISO string of last write. |

**Indexes:**
- `idx_totalScore` — key, attribute `totalScore`, order `DESC` (leaderboard ordering + rank count query).
- `idx_userId` — key, attribute `userId`, order `ASC` (self-lookup for upsert; not unique index — avoids hard 409 on a race).

**Derived (never stored):**
- `rank` — `count of docs with totalScore > ours + 1`, computed at query time via Appwrite `res.total`.
- `accuracy` — `totalQuestions > 0 ? Math.round((totalCorrect / totalQuestions) * 100) : null`. Null maps to `—` in UI.

```js
  bestScore: doc.bestScore,

  totalQuestions: doc.totalQuestions,

```

### `submitScore({ userId, userName, score, correctCount, questionCount })`

Returns `{ success: boolean }`. Never throws — errors are swallowed so a denied or failed write never crashes the result screen.

1. Query existing doc: `listDocuments(config.db, config.col.userStats, [Query.equal("userId", userId), Query.limit(1)])`.
2. **If exists:** `updateDocument(config.db, config.col.userStats, existing.$id, payload)` — no 5th permissions arg (per-doc permissions set at create persist across updates).
   - `totalScore = existing.totalScore + score`
   - `gamesPlayed = existing.gamesPlayed + 1`
   - `bestScore = Math.max(existing.bestScore, score)`
   - `totalCorrect = existing.totalCorrect + correctCount`
   - `totalQuestions = existing.totalQuestions + questionCount`
   - `userName` refreshed, `lastPlayedAt = new Date().toISOString()`
3. **If not exists:** `createDocument(config.db, config.col.userStats, ID.unique(), payload, permissions)`.
   - `payload`: `{ userId, userName, totalScore: score, gamesPlayed: 1, bestScore: score, totalCorrect: correctCount, totalQuestions: questionCount, lastPlayedAt: new Date().toISOString() }`
   - `permissions`: `[Permission.read(Role.any()), Permission.update(Role.user(userId)), Permission.delete(Role.user(userId))]`
   - The 5th-argument permissions array is what enforces per-doc ownership. Requires Document Security ON in the collection (Phase 0 step 2).
4. `try/catch`: on error `console.error("Skor kaydedilemedi:", error); return { success: false };`

---

### `getLeaderboard({ limit = 50, offset = 0 } = {})`

Built now for the future leaderboard screen; no screen is built in this plan.

1. `listDocuments(config.db, config.col.userStats, [Query.orderDesc("totalScore"), Query.limit(limit), Query.offset(offset)])`.
2. `docs.map((d, i) => ({ ...mapDoc(d), rank: offset + i + 1 }))`.
3. `try/catch` → on error `console.error("Liderlik tablosu alınamadı:", error); return [];`

---

### `getUserStats(userId)`

1. Guard: `if (!userId) return null`.
2. `listDocuments(config.db, config.col.userStats, [Query.equal("userId", userId), Query.limit(1)])`.
3. If empty → `return null` (caller shows `—` empty state).
4. Compute `accuracy`: `totalQuestions > 0 ? Math.round((totalCorrect / totalQuestions) * 100) : null`.
5. Compute rank: `listDocuments(config.db, config.col.userStats, [Query.greaterThan("totalScore", existing.totalScore), Query.limit(1)])` → `res.total + 1`. (Appwrite returns accurate `total` count regardless of `limit`.)
6. `return { ...mapDoc(existing), accuracy, rank }`.
7. `try/catch` → on error `console.error("Kullanıcı istatistikleri alınamadı:", error); return null;`

> **Tie-break (V1):** users tied on `totalScore` receive the same rank from `getUserStats`. Documented and acceptable for V1.

## Phase 0 — Appwrite Console Setup (manual prerequisite, no code)

Performed by the user in the Appwrite Console before any code phase runs.

1. In database `expoAppNew`, create collection → **Collection ID:** `userStats`.
2. **Enable Document Security — turn ON.** Without this, the 5th-argument permissions in `createDocument` are silently ignored and per-doc ownership does not work.
3. **Collection-level permissions:** one role → `users/verified`, **Create** only. Do NOT add read/update/delete at collection level — those come from each document's own permissions.
4. Create the 8 attributes exactly as in the Data Model table (types, sizes, required, defaults).
5. Create the two indexes: `idx_totalScore` (totalScore DESC) and `idx_userId` (userId ASC).
6. Verify: with a test verified account, trigger a create → the resulting doc must show `read("any")`, `update("user:<id>")`, `delete("user:<id>")` in the Console.

**Exit criteria:** user confirms all attributes status "available", both indexes built, permissions set, Document Security ON.

## Phase 2 — `quick-game.jsx` game-end score submission

**Goal:** On game completion, call `submitScore` for verified non-guest users. Never write for guests. Never crash on a denied or failed write.

**Steps:**
1. Add imports using relative paths, matching the existing service import convention in this file (line 18 uses `"../../../services/questionService"`):
   ```js
   import { useAuth } from "../../../context/AuthContext";
   import { submitScore } from "../../../services/scoreService";
   ```
2. In component body: `const { user, isGuest } = useAuth();`
3. Add ref near other refs: `const scoreSubmittedRef = useRef(false);`
4. New effect, placed after the existing game-over sound effect:
   ```js
   useEffect(() => {
     if (!gameCompleted || scoreSubmittedRef.current) return;
     scoreSubmittedRef.current = true;
     if (isGuest || !user?.$id) return;   // guests are ephemeral — never write
     if (score <= 0) return;               // skip zero-score games
     const correctCount = score / 10;      // mirrors existing game-end convention
     submitScore({
       userId: user.$id,
       userName: user.name || "Oyuncu",
       score,
       correctCount,
       questionCount: questions.length,
     }).catch(() => {});                   // submitScore already swallows errors; defensive only
   }, [gameCompleted, isGuest, user, score, questions.length]);
   ```
   - `user.$id` read directly from the user object — not added to context.
   - Fire-and-forget: does not block the result screen render.
   - If the server rejects the write (e.g. unverified token), `submitScore` catches, logs, and returns `{ success: false }`. The `.catch` here handles any edge case rejection.
5. In `restartGame`, reset the guard: `scoreSubmittedRef.current = false;` alongside other state resets.

**Validation (device test):**
- Verified user plays a non-zero-score game → `userStats` doc created (first game) or updated (subsequent games) in Appwrite Console with correct cumulative fields and per-doc permissions.
- Guest plays → no doc created or modified.
- `npm run lint` clean.

## Phase 3 — Profile Skor / Doğruluk chips

**Goal:** Replace hardcoded placeholder values with real `totalScore` and accuracy `%N`. Empty state `—` when no doc, guest, or zero games played.

**Steps:**

**`app/(app)/profile.tsx`:**
1. Add imports: `import { getUserStats } from "@/services/scoreService";` and `useEffect` (if not already imported).
2. Add state: `const [stats, setStats] = useState<{ totalScore: number; accuracy: number | null; rank: number } | null>(null);`
3. Add effect on mount:
   ```ts
   useEffect(() => {
     if (isGuest) return;
     const userId = typedUser?.$id as string | undefined;
     if (!userId) return;
     let cancelled = false;
     getUserStats(userId).then((result) => {
       if (!cancelled) setStats(result);
     });
     return () => { cancelled = true; };
   }, [isGuest, typedUser]);
   ```
4. Compute chip values:
   - Skor: `stats ? String(stats.totalScore) : "—"`
   - Doğruluk: `stats?.accuracy != null ? \`%${stats.accuracy}\` : "—"` (`%85` format — Turkish convention)
5. Pass to `ProfileCard`: `scoreValue={scoreValue} accuracyValue={accuracyValue}`.

**`components/ProfileCard.tsx`:**
1. Extend `ProfileCardProps` with `scoreValue?: string` and `accuracyValue?: string`.
2. The chips are currently rendered via a `.map()` over a hardcoded array where `"Yakında"` is in the JSX body, not in the array data. Add a `value` field to each array entry so the map can render per-chip dynamic values:
   ```ts
   const chips = [
     { label: "Skor", value: scoreValue ?? "—", ... },
     { label: "Doğruluk", value: accuracyValue ?? "—", ... },
   ];
   ```
   Then in the `.map()` render, replace the hardcoded `"Yakında"` text with `chip.value`. Keep all existing style/color fields in each array object unchanged.

**Empty-state guarantees (no fake data):**
- Guest → effect returns early, props stay undefined, chips show `—`.
- Verified user with zero games (no doc) → `getUserStats` returns `null`, props stay undefined, chips show `—`.
- `gamesPlayed > 0` but `totalQuestions === 0` (impossible today but guarded) → accuracy is `null` → chip shows `—`.

**Validation (device test):**
- Verified user with game history → Skor shows real `totalScore`, Doğruluk shows `%N`.
- Guest → both chips show `—`.
- New verified user with no games → both chips show `—`.
- `npm run lint` clean; `npx tsc --noEmit` clean.

## Risks and Edge Cases

| Risk | Handling |
|------|----------|
| Upsert race (two near-simultaneous games) | Rare on single-device V1. Read-then-write is not atomic. No unique index (avoids hard 409); worst case = one lost increment, no crash. V2 follow-up: atomic Appwrite Function. |
| Guest plays then converts to a verified account (Layer 2) | Out of scope. Guest games are ephemeral; nothing to migrate. Backlog tracks Layer 2. |
| Score = 0 game | Not submitted (`score <= 0` guard in Phase 2 effect). `gamesPlayed` and accuracy unaffected. |
| `getUserStats` rank via `res.total` | Appwrite returns accurate `total` regardless of `limit`. Verify in Console on first run. |
| Tie on `totalScore` | All tied users get the same rank from `getUserStats`. Documented V1 limitation. |
| Network failure on submit | Silent (logged in service), result screen renders normally. |
| Verified flag unreliable (per backlog) | Server-side `users/verified` collection permission is the real gate. Client guard is a UX optimization. |
| `user.name` empty | Fallback `"Oyuncu"` passed as `userName`. |
| Document Security OFF in Console | Per-doc permissions silently ignored; anyone could update any row. Phase 0 step 2 is mandatory. |
